# Golden Case 3 — 모호한 기획 원문 → 개발자 번역 ('확인 필요' 유도 케이스)

- 방향: `translate(sourceMd, "developer")`
- 목적: 원문이 의도적으로 모호/불완전할 때 번역이 **추정을 본문에 단정하지 않고**
  `> ⚠️ 확인 필요:` 블록으로 분리하는지 검증한다.
- 검증 방법: 아래 원문으로 블록을 보내고, 생성된 번역본을 기대 체크리스트와 수동 대조한다.

## 입력 원문 (기획자 작성 — 의도적으로 모호)

```markdown
## 알림 기능 추가

주문 상태가 바뀌면 사용자에게 알림을 보내주세요.
알림은 너무 자주 가면 안 되고, 하루에 최대 10건 정도면 적당할 것 같습니다.
밤늦게는 보내지 말아 주세요.
중요한 알림은 예외로 하고요.
동시 접속자는 100명 정도를 생각하고 있습니다.
```

## 기대 체크리스트

### 수치·조건 보존
- [ ] `하루 최대 10건`이 그대로 등장하며, 원문의 한정어("정도", "적당할 것 같습니다")가
      확정 요구로 격상되지 않거나, 격상이 필요하면 '확인 필요'로 질문한다
- [ ] `동시 접속자 100명`이 그대로 보존된다 ("정도"의 뉘앙스 포함)

### '확인 필요' 표기 — 이 케이스의 핵심 (기대 위치)
원문이 정하지 않은 아래 항목들은 **본문에 단정되면 실패**, '확인 필요' 블록으로 다뤄지면 통과:
- [ ] "밤늦게"의 구체 시간 범위 (예: 22시~08시 같은 값을 본문에 단정하면 실패)
- [ ] "중요한 알림"의 정의/분류 기준
- [ ] 알림 채널 (푸시/이메일/SMS — 원문에 없음. 특정 채널을 단정하면 실패)
- [ ] 주문 상태의 종류와 어떤 상태 변화가 알림 대상인지
- [ ] 10건 초과 시의 처리(버림/지연/묶음 발송 — 원문에 없음)
- [ ] 최소 2개 이상의 `> ⚠️ 확인 필요:` 블록이 존재한다 (이 원문의 모호도에서 0~1개면
      추정을 본문에 흡수했을 가능성이 큼 — 본문 단정 여부를 재확인)
- [ ] 모든 '확인 필요'가 정확히 `> ⚠️ 확인 필요:` 인용 블록 형식을 따른다

### 무창작
- [ ] 원문에 없는 수치(재시도 횟수, 발송 지연 시간, 큐 크기 등)를 만들어내지 않는다
- [ ] 원문에 없는 기술 결정(특정 메시징 인프라, 우선순위 큐 도입 등)을 요구사항처럼 단정하지 않는다

### 직군 관점 변환
- [ ] 데이터(알림 설정·발송 카운트 등), 흐름(상태 변경 → 발송 판정 → 발송), 예외처리
      (한도 초과·야간 시간대·중요 알림 예외) 관점으로 구조화된다

### 출력 형식
- [ ] 출력은 마크다운 본문만이다 (인사말·메타 문구·전체 코드펜스 없음)
