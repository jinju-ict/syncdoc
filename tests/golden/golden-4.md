# Golden Case 4 — 시니어 개발자 원문 → 입문 기획자 번역 (레벨 맞춤 표현)

- 방향: `translate(sourceMd, "planner", "beginner")`
- 목적: 독자 레벨(입문)에 맞춰 **표현만** 조정되는지 검증한다 — 용어 풀이가 붙되,
  풀이가 원문에 없는 요구사항·수치·정책으로 **창작·격상되면 실패**.
  (레벨 기능의 핵심 리스크: "설명을 덧붙이다 내용을 지어내는 것")

## 테스트 절차

1. **브라우저 A(planner)** 헤더의 **내 레벨**을 `입문`으로 변경한다.
   (작성자인 developer의 레벨은 번역에 영향 없음 — 독자 레벨만 반영된다)
2. **브라우저 B(developer)** 에서 아래 입력 원문을 붙여넣고 **보내기**.
3. dev 서버 터미널에서 `[AI 사용량] translate→planner(beginner) …` 로그를 확인한다
   (op 라벨에 `(beginner)`가 붙어야 레벨이 실제 반영된 것).
4. 브라우저 A 새로고침 → `AI 번역본`을 아래 기대 체크리스트와 대조한다.
5. (선택 — 레벨 대비 확인) 브라우저 A의 레벨을 `전문가`로 바꾸고 **같은 원문을 새 블록으로
   다시 전송**하면 두 번역의 표현 차이를 비교할 수 있다 (완료된 번역은 재생성되지 않으므로
   새 블록으로 비교한다).

## 입력 원문 (시니어 개발자 작성 — 의도적으로 용어 밀도 높음)

```markdown
## 세션 캐시 도입 (DB 부하 절감)

로그인 세션 조회가 전체 DB 읽기 트래픽의 약 60%를 차지한다.
세션 테이블 앞단에 Redis 캐시를 두고 cache-aside 패턴으로 전환한다.
캐시 TTL은 30분으로 설정하고, 세션 변경 이벤트(로그아웃·권한 변경) 발생 시 해당 키를 즉시 무효화한다.
캐시 미스 시에만 DB를 조회하므로 세션 조회 p95 지연이 120ms에서 40ms로 줄어들 것으로 예상한다.
캐시 장애 시에는 캐시를 우회해 DB 직접 조회로 폴백한다(가용성 우선).
```

## 기대 체크리스트

### 수치·조건 보존 (무창작 — 값이 하나라도 바뀌면 실패)
- [ ] `약 60%`(DB 읽기 비중)가 그대로 등장한다
- [ ] 캐시 유지 시간 `30분`이 그대로 등장한다
- [ ] `120ms → 40ms` 개선치가 **"예상"이라는 한정어와 함께** 보존된다 (단정 격상 시 실패)
- [ ] 즉시 무효화 조건(로그아웃·권한 변경)이 누락되지 않는다
- [ ] 장애 시 DB 직접 조회 폴백(가용성 우선)이 누락되지 않는다

### 입문 레벨 표현 — 이 케이스의 핵심
- [ ] 전문 용어(캐시, TTL, cache-aside, 캐시 미스, p95, 폴백 등)가 첫 등장 시
      **괄호 한 줄 풀이** 또는 일상어 대체로 설명된다 (예: "TTL(데이터를 보관하는 유효 시간)")
- [ ] 짧은 문장·단계적 설명으로 구성되어 입문자가 흐름을 따라갈 수 있다
- [ ] "사용자 입장에서 무엇이 좋아지는가"(예: 화면 응답이 빨라짐)가 원문 사실 범위 안에서 설명된다

### 풀이의 창작 격상 금지 (입문 모드 특유의 실패 모드)
- [ ] **TTL 30분을 "30분마다 로그아웃된다"처럼 사용자 동작 정책으로 오역·창작하지 않는다**
      (캐시 유지 시간 ≠ 세션 만료 시간 — 이 혼동이 본문에 단정되면 즉시 실패)
- [ ] 풀이·예시가 원문에 없는 수치(서버 대수, 비용 절감액, 동시 접속자 등)를 만들지 않는다
- [ ] 원문에 없는 예외·정책(특정 사용자 면제, 점검 시간대 등)을 본문에 단정하지 않는다

### '확인 필요' 표기 (기대 위치)
- [ ] 자연스러운 확인 필요 후보(있다면 이 주제여야 함): 30분 기준의 적정성(보안 vs 편의 —
      기획 판단), 무효화 대상 "권한 변경"의 구체 범위, 예상 개선치의 검증 방법 —
      이런 기획 결정 사항이 본문에 단정문으로 들어가 있으면 실패
- [ ] 표기는 반드시 `> ⚠️ 확인 필요:` 인용 블록 형식이다 (입문 레벨에서도 형식 동일)

### 직군 관점 변환 (사용자 가치 / 시나리오)
- [ ] 기술 결정(캐시 도입)이 기획자가 판단할 의미(속도 개선, 장애 시 동작, 보안 영향 후보)로
      정리된다 — 용어를 풀이만 하고 구현 절차를 그대로 나열하면 관점 변환 실패

### 출력 형식
- [ ] 출력은 마크다운 본문만이다 (인사말·메타 문구·전체 코드펜스 없음)

## 판정 메모

- 입문 풀이 항목과 무창작 항목이 충돌해 보일 때의 기준: **용어의 일반적 의미 설명은 허용**,
  **이 시스템이 어떻게 하기로 했는지를 새로 정하면 창작**. (예: "캐시는 자주 쓰는 데이터를
  가까운 곳에 두는 기술" = 허용 / "캐시가 가득 차면 오래된 것부터 지운다" = 원문에 없음 → 창작)
- 판정 불일치 시: 보수적 FAIL 기록 → 정본 모델(gpt-5.1) 재판정이 최종 (플랜 tie-break 규칙).
