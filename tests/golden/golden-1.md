# Golden Case 1 — 기획자 원문 → 개발자 번역 (수치 보존 + 무창작)

- 방향: `translate(sourceMd, "developer")`
- 검증 방법: 아래 원문으로 블록을 보내고, 생성된 번역본을 기대 체크리스트와 수동 대조한다.

## 입력 원문 (기획자 작성)

```markdown
## 포인트 적립 기능

구매 확정 시 결제 금액의 3%를 포인트로 적립합니다.
월 적립 한도는 1인당 5,000P이며, 적립된 포인트는 적립일로부터 90일이 지나면 소멸됩니다.
포인트는 1P = 1원으로 다음 구매 시 사용할 수 있고, 최소 사용 단위는 100P입니다.
구매를 취소하면 해당 구매로 적립된 포인트도 함께 회수합니다.
```

## 기대 체크리스트

### 수치·조건 보존 (무창작 — 값이 하나라도 바뀌면 실패)
- [ ] `3%` 적립률이 그대로 등장한다
- [ ] 월 한도 `5,000P`가 그대로 등장한다 (5000P/5,000포인트 등 표기 변형은 허용, 값 변경은 불가)
- [ ] 소멸 기한 `90일`이 그대로 등장한다
- [ ] 환산율 `1P = 1원`, 최소 사용 단위 `100P`가 그대로 등장한다
- [ ] 취소 시 적립 포인트 회수 조건이 누락되지 않는다

### 무창작 (원문에 없는 내용이 본문에 단정되면 실패)
- [ ] 원문에 없는 수치(예: 소멸 알림 7일 전, 등급별 적립률 등)를 만들어내지 않는다
- [ ] 원문에 없는 정책 결정(예: "포인트는 부분 사용 불가", "환불 시 음수 포인트 허용")을 본문에 단정하지 않는다
- [ ] 원문에 없는 기한·일정을 추가하지 않는다

### '확인 필요' 표기 (기대 위치)
- [ ] 원문이 정하지 않은 사항을 다루려면 반드시 `> ⚠️ 확인 필요:` 인용 블록 형식으로만 표기한다
- [ ] 자연스러운 확인 필요 후보(있다면 이 주제여야 함): 적립 포인트의 소수점 처리 방식,
      포인트가 회수 시점에 이미 사용된 경우의 처리, 적립 시점(구매 확정의 정의) —
      이런 내용이 본문에 단정문으로 들어가 있으면 실패
- [ ] '확인 필요' 블록은 관련 섹션(예: 예외처리) 바로 아래에 위치한다

### 직군 관점 변환
- [ ] 데이터(포인트 잔액·적립 내역 등 엔티티/필드/제약), 흐름(구매 확정→적립→사용→취소 회수),
      예외처리 관점의 구조가 드러난다

### 출력 형식
- [ ] 출력은 마크다운 본문만이다 (인사말·"다음은 번역입니다" 류 메타 문구·전체 코드펜스 없음)
