# Golden Case 2 — 개발자 원문 → 기획자 번역 (기술 제약의 사용자 언어 변환)

- 방향: `translate(sourceMd, "planner")`
- 검증 방법: 아래 원문으로 블록을 보내고, 생성된 번역본을 기대 체크리스트와 수동 대조한다.

## 입력 원문 (개발자 작성)

```markdown
## 검색 API 레이트 리밋 적용

검색 API에 사용자당 분당 30회 요청 제한을 적용한다.
초과 시 429 응답과 함께 Retry-After 헤더(60초)를 반환한다.
클라이언트는 429 수신 시 지수 백오프(최초 1초, 최대 16초)로 재시도한다.
제한 카운터는 Redis에 저장하며 TTL 60초의 슬라이딩 윈도우로 구현한다.
이 변경으로 검색 인프라 비용이 월 약 40% 절감될 것으로 추정된다.
```

## 기대 체크리스트

### 수치·조건 보존 (무창작 — 값이 하나라도 바뀌면 실패)
- [ ] `분당 30회` 제한이 그대로 등장한다
- [ ] 초과 시 `60초` 대기(Retry-After)가 사용자 언어로 보존된다 (예: "1분 뒤 다시 시도")
- [ ] 재시도 정책 수치(`1초`~`16초`)가 왜곡 없이 보존되거나, 생략 시에도 다른 값으로 바뀌지 않는다
- [ ] 비용 절감 `월 약 40%`가 "추정"이라는 원문의 한정어와 함께 보존된다 (단정으로 격상되면 실패)

### 직군 관점 변환 (사용자 가치 / 시나리오)
- [ ] "사용자가 1분에 30회 넘게 검색하면 잠시 검색이 제한된다" 류의 사용자 행동 시나리오로 설명된다
- [ ] 429·Redis·TTL·슬라이딩 윈도우 같은 구현 세부가 기획 판단에 필요한 의미(제한 동작·복구 시간)로
      번역되어 있다 — 기술 용어를 그대로 나열만 하면 관점 변환 실패
- [ ] 사용자 가치/비즈니스 영향(비용 절감, 어뷰징 방지 등 원문 사실 범위 내)이 정리된다

### 무창작
- [ ] 원문에 없는 사용자 영향(예: "일반 사용자의 99%는 영향 없음" 같은 창작 통계)을 만들지 않는다
- [ ] 원문에 없는 예외(VIP 면제, 로그인/비로그인 구분 등)를 본문에 단정하지 않는다

### '확인 필요' 표기 (기대 위치)
- [ ] 자연스러운 확인 필요 후보(있다면 이 주제여야 함): 제한 도달 시 사용자에게 보여줄 안내 문구/UX,
      30회 기준의 적정성(기획 판단), 제한 대상 범위(비로그인 사용자 식별 방법) —
      이런 기획 결정 사항이 본문에 단정문으로 들어가 있으면 실패
- [ ] 표기는 반드시 `> ⚠️ 확인 필요:` 인용 블록 형식이다

### 출력 형식
- [ ] 출력은 마크다운 본문만이다 (인사말·메타 문구·전체 코드펜스 없음)
